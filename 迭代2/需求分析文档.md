# 旅行规划 Agent 需求分析文档

## 项目背景

随着国民旅游消费的持续增长，自由行已成为主流的出行方式。然而，一次完整的旅行规划通常涉及交通、住宿、景点、餐饮等多维度的信息检索与协调编排，用户往往需要在多个平台之间反复跳转、手动比价、自行校验时间与空间的合理性。这一过程不仅耗时耗力，且容易因信息不对称而产生次优决策。

与此同时，大语言模型的成熟，为自动化旅行规划提供了新的技术路径。基于大语言模型的智能 Agent 能够理解用户的自然语言需求，并通过调用外部 API（如地图服务、航班查询、铁路时刻等）获取实时数据，进而生成结构化的旅行计划。通过合理的任务分解与多智能体协作，Agent 可以在保证规划质量的同时显著提升效率和用户体验。

基于此背景，我们设计并实现了一个基于 Multi-Agent 协作的旅行规划系统，通过将旅行规划任务分解为可并行执行的子任务，各 Agent 专注于单一领域，最终汇总生成完整的旅行计划。

## 痛点分析

通过对现有旅行规划方式的观察与分析，我们归纳出以下核心痛点。

### 多平台信息割裂

用户规划一次旅行通常需要在机票预订平台（携程、飞猪）、地图服务（高德、百度）、酒店预订平台、餐饮评价平台（大众点评、小红书）之间反复切换，手动汇总信息。各平台的数据格式不统一，难以进行跨维度的综合比较。

### 时空协调复杂度高

旅行规划本质上是一个带时空约束的组合优化问题：景点 A 的开放时间为 9:00-17:00，景点 B 距离景点 A 有 45 分钟车程，午餐须安排在二者之间，而餐厅 C 在午间需要排队 30 分钟——这些约束相互耦合，人工编排极易遗漏或冲突。

### 实时信息获取滞后

航班准点率、铁路余票数量、酒店实时价格、景点临时闭园等动态信息在传统规划方式中往往被忽略，导致计划与实际执行产生较大偏差。

### 需求表达的“表述-理解”鸿沟

用户的需求往往是模糊和感性的（如“我想体验当地特色美食”“行程不要太赶”），而非结构化的查询参数。传统旅行平台的搜索功能无法有效承接此类自然语言式需求。

### 迭代修改成本高

旅行计划很少一次成型，用户常需根据同行者意见、预算变化或突发情况反复调整。传统方式下，一处修改（如更换酒店）可能引发连锁调整（周边交通、相邻景点），返工成本极高。

## 目标受众分析

本系统的核心受众为自由行人群，即不希望跟团、自主安排行程的旅行者。按照用户的旅行经验和需求复杂度，我们将目标用户细分为以下几类：

### 自由行新手

此类用户对旅行目的地有一定了解，但缺乏从零规划行程的能力。他们在规划时面临的主要困难包括：不清楚如何安排每日行程的时间分配、不了解各景点之间的空间关系、不确定交通方式的选择是否合理。他们期望系统能够根据其出行时间、预算和偏好，自动生成一份可直接执行的行程计划。

### 资深自由行用户

此类用户具备丰富的自由行经验，能够独立完成大部分规划工作。他们的核心诉求是提升旅行规划的效率——希望系统能够快速聚合多源信息、自动校验时空冲突、提供可一键导出至日历或分享给同行者的结构化行程。他们对行程的灵活性要求更高，倾向于在系统生成的框架基础上进行手动微调。

### 偏好驱动的回头客

此类用户已使用系统完成过至少一次旅行规划。系统应能够基于其历史行程与反馈，沉淀用户画像（如偏好景点类型、用餐口味、住宿档次、每日活动强度等），使得后续规划更贴合其个人风格，减少重复表达偏好的摩擦。

## 功能需求

### 行程创建与管理

- 用户应能够创建新的旅行行程，并为每个行程设定基本信息（如目的地、出发地、出行日期、返程日期、同行人数、预算上限等）。
- 用户应能随时查看、修改或删除已创建的行程。
- 行程信息需持久化存储，用户关闭会话后再次打开时仍可恢复之前的规划进度。

### 自然语言交互式规划

- 用户应能够以自然语言向 Agent 描述旅行需求，无需学习特定的查询语法或填写繁琐的表单。
- 系统需理解用户输入中的显性约束（如“预算 3000 元”“不要红眼航班”）和隐性偏好（如“行程不要太赶”），并将这些需求转化为可执行的规划动作。
- 对话过程应以流式方式实时返回规划进度，包括 Agent 当前的推理思路、正在查询的数据源以及已生成的部分结果，避免用户长时间等待无反馈。

### 实时数据查询

规划过程中，系统需能够查询外部实时数据以确保计划的可行性，而非仅依赖 LLM 训练语料中的静态知识。具体包括：

- **航班查询**：根据出发地、目的地和日期，查询可用航班及其时刻、准点率等信息
- **铁路查询**：根据出发地、目的地和日期，查询高铁/动车车次、余票情况和站点信息
- **地理信息查询**：搜索目的地及周边的酒店、景点、餐厅等信息，获取其地理位置（经纬度）、地址、联系方式、营业时间等详细数据
- **路径规划**：查询两地之间的交通方式、距离和预计耗时

### 计划的增量修改

- 用户应能够对已有计划进行局部修改。例如，用户可以要求“把第二天的酒店换成天安门附近的”，系统应仅更新酒店项并自动适配相邻时段的出行路线，而不影响其他日期的已有安排。
- 系统应保留修改历史，支持用户回溯至之前的版本。

### 规划过程的透明化

- 用户在使用 Agent 进行旅行规划时，应能够实时感知系统的执行状态。
- 系统应向用户展示当前处于规划的哪个阶段——是在拆解需求、查询航班、搜索酒店、编排景点还是汇总结果。当系统查询外部数据时，用户应知晓正在查询的数据源和参数。
- 当各维度规划完成后，用户应能看到中间结果。

### 主动引导式参数收集

系统在面对模糊或信息不足的用户输入时，必须具备主动提问的业务逻辑，引导用户补齐规划所必需的关键参数，严禁在参数缺失的情况下依赖模型幻觉直接生成行程。

### 行程中动态调整

系统需支持在用户旅行过程中，基于用户主动上报或系统自动感知，实时处理天气，交通，景点等各方面的突发情况，对后续行程进行实时调整。

### 用户画像与偏好学习

系统应在每次行程结束后，基于用户的实际选择、调整行为与反馈，更新用户画像，使后续规划逐步贴近用户的个人风格。

### 行程分享与导出

行程规划的价值不仅在于制定，还在于执行过程中的便捷使用和与他人分享。

- **行程分享**：系统应支持生成行程分享链接，接收方无需登录即可查看行程概览（含每日行程安排、景点信息、交通衔接等），并支持设置分享权限（仅查看/允许评论）。
- **图文卡片导出**：系统应支持将每日行程导出为图文卡片（图片格式），内容包含景点名称、地址、开放时间、交通建议及地图示意，适合在微信、小红书等社交平台分享或离线查看。
- **日历同步**：系统应支持将行程中的关键节点（航班/火车出发到达、酒店入住退房、景点预约时间等）一键同步至用户手机日历（支持 Apple Calendar / Google Calendar / 国产手机原生日历），自动设置提醒。
- **离线访问**：导出的图文卡片和日历事件应在无网络环境下仍可正常查看，确保用户在旅途中（如飞行模式、境外无网络）仍能获取行程信息。

## 非功能性需求

### 性能

- 单次规划请求的完整响应时间应控制在 30 秒以内（不含 LLM API 的网络延迟波动）
- SSE 首字节响应时间应控制在 500ms 以内
- 系统应支持至少 10 个并发规划请求

### 可用性要求

- 当 Agent 在执行过程中遇到故障（如外部 API 调用失败、LLM 响应异常等）时，应能够捕获异常并向用户提供清晰的错误提示，而非直接崩溃或无响应

### 安全性要求

- 所有请求必须携带 Token 进行身份鉴权
- 用户数据进行请求级隔离，不同用户会话之间不得发生数据串扰

### 可扩展性

- 新增 Agent 只需增加配置文件，无需修改核心框架代码
- 更改 Agent 的 Prompt 只需更改配置文件，无需修改代码
- 新增外部工具只需实现接口，框架层自动感知
