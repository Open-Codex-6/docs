# 旅游规划 Agent 评估方案

## 一、核心评估指标体系

| 指标                       | 意义                                                         | 实现方式                                                     | 需要的数据                           |
| :------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------- |
| **交付率 (DR)**            | Agent 能否生成格式正确、可解析的行程单。是最基础的可用性门槛。 | 检查输出是否为结构化 JSON/表格，字段齐全且能被解析。格式错误即判为 0。 | Agent 输出文本                       |
| **环境可行性 (EPR)**       | 行程是否违反物理世界规则（如营业时间、交通方式、地理位置等）。防止“虚构世界”方案。 | 解析行程后，逐项校验：**POI** 是否存在、营业时间是否合理、交通方式/班次是否存在、通勤耗时是否合理。任一失败即为不可行。 | 行程 JSON、景点/酒店/餐厅/交通数据库 |
| **逻辑约束满足率 (LPR)**   | 是否满足用户明确提出的硬性要求（如预算、指定 POI、酒店设施、菜系等）。 | 用户需求转为 **DSL** 校验函数（如 `total_cost(plan) <= budget`），逐条校验。 | 用户需求、行程 JSON、DSL 校验代码    |
| **条件逻辑通过率 (C-LPR)** | 防作弊评估：仅在环境可行时才计入逻辑满足，避免“假满足”。     | 先判 EPR 通过，再统计 LPR 满足项占比。                       | 同上                                 |
| **软偏好对齐度**           | 对“多逛景点”“少走路”等非强制偏好的量化比较。用于横向对比方案优劣。 | 提取如景点数、步行时长等数值，与基线方案排序对比，输出偏好对齐报告。 | 行程 JSON、偏好指标提取函数          |
| **Token 消耗 (TC)**        | 衡量单次任务资源成本，支持模型/提示词/工具链的成本对比。      | 记录 `prompt_tokens`、`completion_tokens`、`total_tokens`，统计均值、P95、最大值。 | Agent 运行日志、模型 usage 字段      |
| **响应时间 (RT)**          | 衡量用户可感知时延，反映系统交互性能与工程可用性。             | 记录首 token 时间（TTFT）与端到端完成时间（E2E latency），统计均值、P95、超时率。 | 请求时间戳、流式事件时间戳、完成时间 |

> **说明**：C-LPR 是区分“纯 LLM”与“带规划模块”Agent 的关键指标，能有效反映系统真实能力。
>
> POI : point of interest 可以理解为旅行中你要去的具体地方，系统需要知道 Agent 安排的这些 POI **是否存在**、**营业时间对不对**、**价格是否真实**
>
> DSL : Domain-Specific Language 专门为“旅游规划检查”设计的一套**简易编程规则**，它能把用户说的人话自动翻译成电脑能执行的计算逻辑。例如：**用户需求**：“酒店要有棋牌室”，DSL:
>
> ```
> 遍历行程中的所有住宿活动
> → 提取酒店的“设施标签”
> → 判断标签里是否包含“棋牌室”
> → 返回 是/否
> ```
>
> - **用户需求**：“总花费不超过 5000 元”
>
> - **DSL 逻辑**：
>
>   ```
>   所有景点门票 + 所有餐厅消费 + 所有酒店房费 + 所有交通费
>   → 求和
>   → 判断是否 ≤ 5000
>   ```

## 二、Ragas 评估指标补充（很多需要LLM-as-Judge）

| 指标                                                         | 意义                             | 实现方式                                               | 需要的数据                                                   | 典型场景                 |
| :----------------------------------------------------------- | :------------------------------- | :----------------------------------------------------- | :----------------------------------------------------------- | :----------------------- |
| **目标达成率**AgentGoalAccuracy                             | Agent 是否真正完成了用户目标     | 多轮对话，LLM 自动推断终态与目标比对                   | **完整对话消息序列**（用户与 Agent 的所有对话内容，包括工具调用/回复等）、**人工撰写的目标描述**（如“成功预订到指定酒店”） | 复杂行程、动态需求       |
| **工具调用正确率**ToolCallAccuracy/F1                       | 工具链是否按预期调用且参数正确   | 对比 Agent 实际调用的工具及参数与“理想调用序列”        | **完整对话消息序列**（需包含每次工具调用的名称和参数）、**理想工具调用清单**（人工标注或自动生成） | 需要外部 API/检索/订票等 |
| **主题边界遵守** TopicAdherence                              | 是否聚焦旅游域并拒答越界问题     | 注入越界问题，统计拒答与主题内答复准确率               | **对话消息序列**（含主题内与主题外问题）、**主题分类标签**（如“旅游/非旅游”） | 防止“答非所问”或安全风险 |
| **检索相关性/精度/召回**ContextRelevance/Precision/Recall/EntityRecall | 检索内容是否相关、有用、覆盖充分 | 结合用户提问、Agent 检索到的内容、人工参考答案进行比对 | **用户提问文本**、**Agent 检索到的文档内容**、**人工参考答案或知识点** | RAG 场景                 |
| **回答正确性/相关性**AnswerAccuracy/FactualCorrectness/AnswerRelevancy | 行程建议本身是否正确、贴题       | 参考答案对比、事实核查、相关性打分                     | **用户提问文本**、**Agent 生成的行程文本/回复**、**人工参考答案** | 关键决策、用户信任       |
| **证据一致性**Faithfulness/ResponseGroundedness/QuotedSpansAlignment | 回复是否“有据可依”且不幻觉       | 拆分陈述，核查是否被检索内容支持                       | **用户提问文本**、**Agent 生成的行程文本/回复**、**Agent 检索到的文档内容** | 可信度要求高             |
| **噪声鲁棒性**NoiseSensitivity                              | 加入干扰文档后是否被误导         | 对比干净/有噪声检索下的错误率                          | **用户提问文本**、**Agent 生成的行程文本/回复**、**人工参考答案**、**干净与含噪声的检索内容** | 检索系统健壮性           |
| **结构化可执行性/业务约束**DataCompyScore + Rubrics         | 行程结构是否可落地、约束是否满足 | 行程转表格/JSON，硬约束用 DataCompy，软约束用 Rubrics  | **Agent 输出的结构化行程（JSON/表格）**、**人工标准答案（JSON/表格）**、**约束描述/评价标准** | 复杂业务场景             |

## 三、Benchmark 构建与实现方法

### 评估流程

1. **输出解析**  
   - 检查 Agent 输出格式，能否被解析为结构化数据（JSON/表格）→ 计算 DR。
2. **环境可行性校验**  
   - 用沙盒数据库校验 POI、时间、交通等 → 计算 EPR。
3. **逻辑约束校验**  
   - 用户需求转 DSL 校验，统计满足项 → 计算 LPR。
4. **条件逻辑通过率**  
   - 仅在 EPR 通过时统计 LPR → 计算 C-LPR。
5. **软偏好评估**  
   - 提取如景点数、步行时长等指标，与基线方案排序对比。
6. **Ragas 指标评测**  
   - 按需补充目标达成率、工具调用、检索/回答/证据等指标，综合分析。
7. **性能评测**
   - 统计 Token 消耗（prompt/completion/total）与响应时间（TTFT/E2E），输出成本-性能对比报表。
