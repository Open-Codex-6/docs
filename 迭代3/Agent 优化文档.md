# 旅行规划 Agent 优化报告

## 1. 背景与问题分析

### 1.1 项目概况

本应用采用 Supervisor–Workers 多智能体协作架构。Supervisor 作为行程规划主管负责全局统筹与任务分发，下设长途交通、市内交通、酒店、景点、餐饮五类专业 Worker，各 Worker 通过调用高德地图、航班查询、铁路查询等外部工具获取真实数据，给出行程建议，最终由 Supervisor 汇总各 Worker 的输出并生成完整的旅行计划。

### 1.2 优化前核心痛点

基于对 102 个真实样本的系统性审查，我们识别出以下三类核心问题：

| 问题类别 | 具体表现 | 根因分析 |
|---------|---------|---------|
| **POI 幻觉与数据缺失** | 经纬度值为零或缺失；`poi_id` 由 LLM 自行虚构（如填充 "12345"）；经度与纬度字段颠倒；地址信息仅精确至城市级别（如"北京市"） | System Prompt 未对 LLM 施加"数据必须来源于工具返回值"的强制性约束 |
| **时间安排冲突** | 同一日内多个计划项的时间区间存在显著重叠，且未预留合理的缓冲时间 | 行程规划流程缺少市内出行规划环节，各景点间的交通耗时未被纳入时间轴计算；各agent并行计算，未能实时更新上下文 |
| **路线顺直度不足** | 一日内景点游览顺序不合理，存在折返与绕行、 | 路线编排未基于真实地理坐标进行优化，依赖 LLM 的常识性判断而非精确空间计算 |

### 1.3 评估指标及优化前基线

针对以上的痛点，我们设计了两个指标。

| 评估指标 | 含义 | 优化前均值 | 含义解读 |
|------|-----------|------|------|
| Location Reliability | 评估行程中地点的真实性 | 0.687 | Agent 提供的约 31.3% 的地点存在幻觉或偏移 |
| Schedule Reasonability | 评估行程在时间上是否存在冲突 | 0.156 | 约 85% 的行程存在不同程度的时间冲突 |
| Route Efficiency | 评估行程路线在空间上是否顺路 | 0.575 | 路线顺直度仅约 57.5%，存在显著优化空间 |

## 2. 总体优化策略

本次优化的目标是解决上一节中识别出的三类核心痛点。

我们沿四个方向展开系统性优化：**上下文工程**从 Prompt 约束与上下文构造层面治理 POI 幻觉与决策失序，同时将高频场景的最佳实践与领域知识编码为可按需加载的技能 SOP；**Agent 架构优化**通过补充市内出行专职 Agent 填补规划链路的结构性盲区；**知识增强**引入实时网页搜索拓展信息获取边界；**工具链重构**通过角色化工具分配与写入时地理编码校验与自动修复构建质量闭环。四个方向从生成端到验证端协同覆盖全链路，共同驱动各项评估指标的提升。

| 优化方向 | 针对痛点 | 核心策略 | 目标指标 |
|---------|---------|---------|---------|
| 上下文工程 | POI 幻觉与数据缺失；时间冲突 | 坐标来源硬约束、技能 SOP 与领域知识注入、全局上下文注入 | Location Reliability, Schedule Reasonability |
| Agent 架构优化 | 时间冲突、路线失序 | 新增 `LocalTransportAgent` | Schedule Reasonability, Route Efficiency |
| 知识增强 | 信息时效性不足 | 网页实时搜索 | 规划方案信息质量 |
| 工具链重构 | 工具误用风险、数据验证缺失 | 角色化工具分配、写入时地理编码校验与自动修复 | Location Reliability |

## 3. 方向一：上下文工程

### 3.1 提示词工程

#### 痛点

POI 幻觉与数据缺失是优化前最突出的质量问题。各 Agent 的 System Prompt 仅以字段描述的形式定义了输出的数据结构（如 `location: {poi_id, poi_name, address, lng, lat}`），但并未对字段值的来源做出任何约束。当高德地图返回的数据中缺少某些字段时，LLM 会基于其语言模型的生成特性，自然地"填补"这些空缺——`poi_id` 被填充为形如 "12345" 的虚构标识符，经度与纬度字段发生颠倒，地址信息仅精确至城市级别（如"北京市"）。

这些问题的根本原因并非 LLM 能力不足，而在于 System Prompt 中缺少一条明确的指令：**结构化数据必须来源于工具返回值，不得由模型自行补全或编造**。

#### 优化方案

在所有 Agent（Supervisor 与全部 5 个 Worker）的 System Prompt 中，对 `location` 的四个关键字段施加统一的、措辞一致的硬约束：

- `poi_id` 必须使用高德地图接口返回的 poi_id，严禁自行编造，若高德地图未返回该字段则保持为空值
- `lng` / `lat` 必须使用高德地图接口返回的坐标值，Prompt 中明确标注"lng 表示经度，lat 表示纬度"以针对性消除坐标颠倒错误
- `address` 必须使用高德地图接口返回的地址文本，力求精确至门牌号级别，严禁自行编造。该约束通过**双重覆盖**机制确保执行：Supervisor 的 System Prompt 中在列举各 Worker 职责时即纳入此约束
- 每个 Worker 自身的 System Prompt 中再次以相同措辞强调。无论模型从哪条指令路径理解任务，均会接收到该规则。

#### 效果

通过上述硬约束的引入，POI 数据质量问题得到了显著缓解。`poi_id` 的输出值为高德地图官方标识符或空值，不再出现虚构标识；经纬度坐标值直接来源于高德 API 响应，不再出现 (0, 0) 空值或经纬度颠倒的错误；地址精度从城市级别的模糊描述（如"北京市"）提升至街道门牌号级别的精确地址（如"北京市东城区景山前街 4 号"）。从生成源头消除了 POI 幻觉的产生条件，很大程度地缓解了 POI 数据质量问题。

### 3.2 技能系统：将最佳实践与领域知识编码为可注入的 SOP

#### 痛点

时间冲突与路线不合理的问题，部分根源于 Supervisor 在面对复杂规划任务时缺乏结构化的操作指南。面对"新行程规划""行程修改""预算优化""突发情况处理"等高频场景，Supervisor 仅依赖通用的 System Prompt 指导，缺少针对具体场景的标准化操作流程，导致规划过程中容易遗漏关键步骤（如景点间预留交通缓冲时间、修改操作的影响范围控制等），进而引发时间冲突与方案质量不稳定。此外，Agent 在处理预算优化、行程修改等专业场景时，也缺乏结构化的领域专业知识作为决策依据——例如在进行预算拆解时，LLM 仅凭通用常识给出粗略的分配建议，缺少各品类（交通、住宿、餐饮等）的合理比例参考；在推荐成本压缩方案时，缺少针对不同消费品类的具体优化策略，导致规划方案的专业性与实用性不足。

#### 优化方案

我们构建了一套基于 YAML 的技能注册表（SkillRegistry），将高频场景的标准化操作流程与结构化领域知识以可注入 Prompt 的形式固化。具体包含四项技能：

| 技能标识 | 功能定位 |
|---------|---------|
| `new_trip_planning` | 新行程规划的完整 SOP：前置信息收集 → 并行委托策略 → 时间轴合理性校验（要求景点间预留 30–60 分钟交通时间） |
| `itinerary_modification` | 行程增量修改：限定修改范围，确保局部调整不波及全局已确认计划 |
| `budget_optimization` | 预算优化规划：提供预算拆解参考比例与各品类成本压缩策略 |
| `emergency_handling` | 突发情况应对：航班取消、景点闭园、极端天气等意外事件的处置流程 |

每份技能的 YAML 配置文件不仅定义了操作流程，更承载了结构化的领域知识。以 `budget_optimization.yaml` 为例，其内容涵盖：预算拆解参考比例（长途交通 25%–35%、住宿 25%–35%、餐饮 15%–20% 等）、各品类的成本压缩策略（经济舱/二等座优先、免费景点替代方案等）以及与用户画像字段的联动规则。技能元数据通过 `{skills_section}` 占位符动态注入 Supervisor 的 System Prompt；当 Supervisor 判定当前场景匹配某项技能时，可调用 `load_skill` 工具获取完整执行指南与领域知识。这一按需加载机制避免了将所有技能内容一次性注入上下文所导致的 Token 膨胀与注意力稀释。

#### 效果

技能系统的引入使 Supervisor 在高频场景下的决策过程从"依赖通用指令的自由发挥"转变为"遵循结构化 SOP 的规范执行"。`new_trip_planning` 技能直接作用于 Schedule Reasonability 指标，通过强制要求时间轴校验显著降低了时间冲突的发生率；`itinerary_modification` 技能有效防止了修改操作引入新的时间冲突；`budget_optimization` 技能提升了规划方案的经济可行性；`emergency_handling` 技能增强了系统在面对突发情况时的鲁棒性。同时，结构化领域知识的注入使规划方案从"基于通用常识的粗略建议"升级为"基于专业知识的精准方案"，YAML 格式的知识配置也便于持续迭代更新，为未来知识库的扩展提供了良好的架构基础。

### 3.3 子 Agent 全局上下文注入

#### 痛点

优化前，Worker Agent 仅接收 Supervisor 下发的一行简短任务描述，缺乏对用户原始需求与整体计划状态的感知。这种信息不对称导致各 Worker 在规划本领域内容时，容易产生与全局脱节的局部决策——例如景点 Agent 安排的游览时间与餐饮 Agent 推荐的用餐时间产生重叠，或酒店 Agent 推荐的住宿位置与当日景点分布存在显著空间偏离。各 Agent 并行计算时无法实时感知彼此的输出状态，进一步加剧了时间区间重叠的问题。

#### 优化方案

在每个 Worker 的 Task Prompt 中注入四项全局上下文：`user_request`（用户原始需求的完整文本，确保 Worker 理解用户的真实意图）、`plan_snapshot`（当前计划的 JSON 快照，包含已确认的全部计划项及其时间安排）、`date_hint`（日期提示，明确当前规划所涉及的具体日期范围）以及格式化后的用户画像摘要（包含用户的出行偏好、预算范围、特殊需求等个性化信息）。通过这一机制，每个 Worker 在接收到具体任务时，同时获得了完整的全局视野。

#### 效果

全局上下文注入使得各 Worker 在规划本领域内容时，能够感知完整计划状态与用户的个性化偏好，有效避免了因信息不对称导致的时间区间重叠。例如，景点 Agent 在安排游览计划时可参考已确认的餐饮时间段，餐饮 Agent 在推荐用餐地点时可参考当日景点的地理分布。这一优化直接贡献于 Schedule Reasonability 指标的提升，是时间冲突问题得到根本性改善的重要驱动因素之一。

## 4. 方向二：Agent 架构优化——填补市内出行规划空白

### 4.1 新增 LocalTransportAgent

#### 痛点

优化前的 Agent 架构中，景点之间的市内交通（公交、地铁、打车、步行等）处于无人负责的真空地带：Supervisor 仅承担统筹角色，Traffic Agent 的职责范围限定于长途交通（航班与铁路）。这一架构缺陷直接导致了两类问题：一是**时间冲突**——各计划项之间缺少交通耗时的真实预估，被紧密排列在时间轴上，Schedule Reasonability 仅为 0.156；二是**路线失序**——游览顺序完全由 LLM 基于常识判断排列，未考虑地理空间距离，Route Efficiency 仅为 0.575。市内出行规划的架构盲区是上述两项指标低迷的核心根因。

#### 优化方案

新增 `LocalTransportAgent`，专职负责城市内部的短途出行规划。该 Agent 利用高德地图的真实路线数据，为每段行程产出结构化的出行方案（包含路线数组、预估耗时、交通费用及起止点坐标）。优化后的 Agent 协作拓扑如下：

```
Supervisor（行程规划主管）
  ├── Traffic（长途交通：航班/铁路）
  ├── LocalTransport（市内出行：公交/地铁/打车/步行）  ← 新增
  ├── Hotel（酒店住宿）
  ├── Attraction（景点游玩）
  └── Food（美食餐饮）
```

各 Worker 保持严格的单一职责，"市内出行"从原先的职责模糊状态中被剥离为独立的规划单元，由专职 Agent 基于高德地图的真实路径数据进行方案产出，使得时间预估与路线排序均建立在可验证的数据基础之上。

#### 效果

`LocalTransportAgent` 的引入是 Schedule Reasonability 与 Route Efficiency 两项指标显著改善的关键驱动因素。在时间维度上，该 Agent 为景点间移动提供了基于真实路径数据的交通耗时预估，原则上预留 30–60 分钟缓冲时间，填补了优化前时间轴计算中的关键缺失项，使行程的时间安排合理性得到根本性改善。在空间维度上，基于高德地图的真实路线数据而非 LLM 的地理常识推断规划景点间移动，使路线排序具有了空间计算依据，显著提升游览顺序的合理性。

## 5. 方向三：知识增强

传统的 RAG 只能从静态知识库中检索信息，无法满足旅行规划中对信息时效性与准确性的要求。因此，我们增加了实时搜索工具，相较于传统 RAG，能够在规划过程中主动调用互联网搜索引擎获取最新信息，显著提升规划方案的时效性与准确性。

### 5.1 网页实时搜索能力引入

#### 痛点

优化前，Agent 的信息检索范围局限于高德地图 POI 数据与航班/铁路时刻表两类结构化数据源。当用户提出"查询景点近期活动""获取酒店真实住客评价""了解当地美食排行榜"等具有时效性要求的信息需求时，LLM 仅能依赖其训练数据中的静态知识进行回答，无法保证信息的准确性与时效性，可能导致推荐已关闭的景点、提供过时的价格信息或遗漏当季特色活动等问题。

#### 优化方案

集成了基于 Tavily 的网页搜索引擎（`websearch_` MCP 工具），并将该工具分配给全部 Worker Agent。每位领域专家在处理其专业领域的任务时，可主动发起互联网搜索以获取最新信息——景点 Agent 可检索景区最新公告与活动安排，酒店 Agent 可查询真实住客评价与最新房价，餐饮 Agent 可获取当地美食排行榜与餐厅营业状态。搜索工具与高德地图等结构化数据源形成互补，构建了"结构化数据 + 实时网页信息"的双层信息获取架构。

#### 效果

实时网页搜索能力的引入显著拓展了 Agent 的信息获取边界，使其从仅依赖静态结构化数据源升级为具备实时互联网检索能力的智能规划系统。规划方案中的信息时效性与准确性得到了显著提升，能够为用户提供包含最新活动信息、真实评价反馈与实时价格数据的高质量行程建议，有效避免了因信息过时而导致的规划失误。

## 6. 方向四：工具链重构

### 6.1 工具按角色精确分配

#### 痛点

优化前，各 Agent 的工具访问权限未进行严格隔离，存在工具误用的风险。例如，景点 Agent 可能错误调用航班查询工具，或餐饮 Agent 可能尝试使用铁路查询接口，导致无效的工具调用浪费计算资源并可能引入错误结果。同时，过大的工具选择空间增加了 LLM 的决策负担——当面对十余种可用工具时，模型需要消耗更多的推理能力来判断当前任务应使用哪个工具，降低了工具选择的准确率。

#### 优化方案

通过 `tool_allocation.yaml` 实现了基于角色的细粒度工具权限控制。Supervisor 持有 `save_*` 与 `delegate_*` 系列工具，专注于计划保存与任务委派；各 Worker 仅被授予其职责领域所需的 MCP 工具前缀——Traffic 持有 `variflight_` 与 `12306_` 用于航班与铁路查询，LocalTransport 持有 `amap_` 系列用于市内路线规划，Hotel、Attraction、Food 等 Worker 持有 `amap_` 与 `websearch_` 用于 POI 检索与实时信息查询。每个 Agent 的工具集合与其职责边界严格对齐。

#### 效果

精确的工具边界约束有效降低了误用风险，杜绝了跨领域的无效工具调用。同时，每个 Agent 面对的工具选择空间大幅缩小，使得 LLM 在工具选择环节的推理负担显著降低，工具调用的准确率与效率均得到提升。这一优化还带来了额外的安全性收益——各 Agent 仅能访问其职责范围内的数据源，遵循了最小权限原则。

### 6.2 写入计划工具中校验并自动修复地理数据

#### 痛点

优化前，虽然 Agent 的 System Prompt 中已要求使用高德地图数据，但在 Agent 输出结果写入旅行计划的环节缺乏系统性的校验机制。即使在 Prompt 层面施加了坐标来源约束，仍无法完全排除 LLM 在生成过程中出现的数据偏差——例如坐标值的微小偏移、POI 名称与实际地点不匹配等问题。单纯依赖 Prompt 约束的"单侧防线"意味着错误数据一旦生成即直接落库，缺少在写入前拦截与修正的机会。

#### 优化方案

在 Agent 侧 Prompt 约束（"坐标必须来源于高德地图返回值"）的基础上，在旅行计划的写入工具（`save` 工具）中嵌入了三级地理编码校验与自动修复流水线：第一级为**基础完整性检查**，验证坐标字段是否非空、非零且在合法范围内；第二级为**高德地理编码反查**，以 POI 名称为查询词调用高德地理编码 API，验证该 POI 是否真实存在；第三级为**坐标偏移量检测**，计算 Agent 输出坐标与高德基准坐标之间的 Haversine 距离，判定偏移是否超过阈值。当校验发现数据异常时，写入工具自动使用高德 API 返回的正确数据进行覆盖修复，确保最终落库的 POI 数据均为经过验证的可靠数据。

#### 效果

由此构建了"Prompt 约束 → 高德返回真实数据 → 写入时二次校验与自动修复"的完整防幻觉闭环，从生成端与写入端双向保障 POI 数据的可信度。与事后评估标记不同，该机制在数据落库前即完成错误拦截与修正，确保旅行计划中存储的每一个 POI 坐标均为经过高德地理编码验证的可靠数据，从根本上杜绝了幻觉数据流入下游地图渲染与导航功能的可能性。

## 7. 优化效果分析

| 评估指标 | 改善前均值 | 改善后均值 | 提升幅度(均值之差) |
|------|--------|--------|---------|
| Location Reliability | 0.687 | **0.875** | **+18.8%** |
| Schedule Reasonability | 0.156 | **0.824** | **+66.8%** |
| Route Efficiency | 0.575 | **0.801** | **+22.6%** |

### Location Reliability 指标分析

该指标从 0.687 提升至 0.875（+16.8%），表明行程中地点数据的真实性与准确性得到了显著改善。优化前，经纬度为零或缺失、poi_id 被 LLM 虚构、lat/lng 颠倒等数据质量问题普遍存在。提升主要来源于两层机制的协同作用：

- **生成端的 Prompt 硬约束**：对全部 Agent 施加统一的坐标数据来源硬约束，从源头消除了 POI 幻觉的产生条件——`poi_id` 输出值为高德地图官方标识符或空值，不再出现虚构标识；经纬度坐标直接来源于高德 API 响应，不再出现 (0, 0) 空值或经纬度颠倒的错误；地址精度从城市级别的模糊描述（如"北京市"）提升至街道门牌号级别（如"北京市东城区景山前街 4 号"）；
- **写入端的三级校验与自动修复**：写入工具在数据落库前执行完整性检查、高德地理编码反查与坐标偏移量检测，当发现异常时自动使用高德 API 返回的正确数据进行覆盖修复，确保最终存储的每一个 POI 坐标均经过验证，为下游地图渲染与导航功能提供了可信的数据基础。

### Schedule Reasonability 指标分析

该指标从 0.156 提升至 0.824，意味着行程的时间安排合理性从"仅约 15% 的样本无冲突"扭转为"超过 82% 的样本满足时间约束"。主要驱动因素包括：

- **LocalTransportAgent 的引入**：为景点间移动提供了基于真实路径数据的交通耗时预估（原则上预留 30–60 分钟缓冲时间），填补了优化前时间轴计算中的关键缺失项；
- **新行程规划技能的约束**：`new_trip_planning` 技能明确要求 Supervisor 在汇总各 Worker 结果后逐一校验时间轴的合理性；
- **子 Agent 全局上下文的注入**：各 Worker 在规划自身领域时可感知完整计划状态，避免因信息不对称导致的时间区间重叠。

### Route Efficiency 指标分析

路线顺直度从 0.575 提升至 0.801，表明游览顺序的合理性获得了显著改善。提升来源于两个层面：

- **高德真实路径数据的采用**：LocalTransportAgent 基于高德地图的真实路线数据（而非 LLM 的地理常识推断）规划景点间移动，使路线排序具有了空间计算依据；
- **坐标数据准确性的保障**：Prompt 中的坐标来源硬约束消除了 lat/lng 颠倒与空值问题，确保路线距离计算建立在正确的经纬度基础之上。
